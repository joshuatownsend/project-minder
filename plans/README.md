## Findings considered and rejected

Recorded so they aren't re-audited next run. Each was opened and verified against the code.

- **Unsafe `yaml.load` / prototype pollution** (`parseFrontmatter.ts`, `gsdPlanning.ts`,
  `memory.ts`): js-yaml **4.1.1** — `load()` *is* the safe loader (no `!!js/*` types in the
  default schema, drops `__proto__`); `safeLoad` was removed in v4. `parseFrontmatter` also
  only returns plain objects. Not a vulnerability.
- **`costCalculator.ts` untested**: false — `tests/usage/costCalculator.test.ts` exists.
- **N+1 prepared statements in `db/ingest.ts`**: statements are prepared once (lines
  1295–1336) and reused in the row loop (line 1357). Per-session re-prepare is micro-opt at
  most, not the algorithmic N+1 reported. Not worth doing.
- **tsconfig `incremental: true` masks cross-file type errors**: already mitigated —
  `scripts/typecheck.mjs` deletes `tsconfig.tsbuildinfo` before *every* run (local,
  pre-commit, CI). The dirty `tsconfig.json` in the working tree at audit time was an
  uncommitted experiment re-adding the flag; HEAD has none.
- **`better-sqlite3` absence crashes the server**: false — `db/connection.ts:36–44` wraps
  the `require` in try/catch and records `loadError`; degrades gracefully.
- **React memoization / list virtualization on big components**: speculative, cited wrong
  paths (`src/app/components/` — the dir is `src/components/`), no evidence the lists are
  large/janky, and virtualization is already applied where it matters (SparklineList). Not a
  vetted finding.
- **Client-side fetch waterfalls / convert pages to Server Components**: by-design — all
  pages are `"use client"` per the repo's stated conventions. Architectural rewrite, not a fix.
- **God-module refactors of `data/index.ts` (1,044 LOC) and `UsageDashboard.tsx` (1,278 LOC)**:
  high blast radius, low payoff; the size alone isn't a defect and there's no churn/feature
  pressure forcing it. Defer until a concrete need (e.g. a third data adapter) arises.
- **`jsonlWatcher` doesn't close the watcher in its error path** (`agentView/jsonlWatcher.ts:76`):
  real but marginal (handle leak only if recursive-watch setup throws). LOW impact; not
  selected for a plan this round — revisit if watcher leaks are observed.
- **`aggregator.ts` orchestration untested** (`usage/aggregator.ts`): a real coverage gap
  (sub-modules are tested individually; the composition isn't). Surfaced to the user but not
  selected for a plan this round; a future characterization test over `generateUsageReport`
  with fixture sessions would lock the contract.

  - **`configHistory` module** (`src/lib/configHistory.ts`) — `recordPreWrite(targetPath, opts?)`, `restore(backupId)`, `list({projectSlug?})`, `prune(now?)`. Snapshots write to `~/.minder/config-history/<id>/<basename>` (base64-encoded bytes), manifest at `~/.minder/config-history/manifest.jsonl`. BackupId format `<iso-ts>_<sha8|"missing">_<rand4>` (random suffix added in the PR-#59 review-fix commit; pre-fix format `<iso-ts>_<sha8>` could collide on same-ms same-content recordings). Each `recordPreWrite` records a fresh entry — no dedup-on-write; the smart-retention prune drops duplicates over time. `recordPreWrite` is fail-open: a snapshot failure logs a warning and returns null but never blocks the apply (a missing backup is unfortunate, an aborted apply is worse). Smart-retention `prune()` per TODO #56: every entry within 24h, daily for 7d, weekly for 30d, drop older. Auto-prune triggered opportunistically from `recordPreWrite` when last-prune timestamp (sidecar `.last-prune` file) is > 1h old.
  - **Apply-layer wiring** (`src/lib/template/apply.ts`) — every dispatch helper that writes a single file (agent, standalone skill, command, hook, MCP, workflow, settings-key, plugin enable) calls `snapshotBeforeApply()` with the target file path before invoking the primitive. Bundled-skill applies (which write a directory tree) intentionally pass through unwrapped — directory snapshots are a separate retention problem and out of Wave 1.2 scope. Dry-run applies never record. Snapshot label is `apply-<unitKind>:<unitKey>` so the Config History tab can show what triggered each entry.
  - **Config History tab** on `ProjectDetail` — `TabKey` union extended with `"config-history"`. Tab appears only when `/api/config-history?project=<slug>` returns at least one entry (lazily fetched on mount). `<ConfigHistoryTab>` component lists timestamp / target path / label / SHA-prefix / restore button per entry; restore round-trips through `POST /api/config-history/restore` and refreshes the list. `useToast` for success/failure. New help doc `docs/help/config-history.md` (+ runtime copy) describes the snapshot scope and retention policy.
  - **API routes** — `GET /api/config-history?project=<slug>` returns the manifest entries (filtered by project), `POST /api/config-history/restore` validates the BackupId via `list()` before calling `restore()` so a bad ID surfaces 404 rather than 500. Restore invalidates scan cache, claude-config route cache, and userConfigCache so the dashboard reflects the rolled-back file immediately.
  - **`ensureSchemaReady` 30s failure-TTL** (`src/lib/data/index.ts`) — replaces P2c's "clear cache on every failure" with a uniform 30s TTL on cached failures. P2c fixed the original bug (cached forever was poisoning every subsequent read) but introduced a new one: every dashboard poll re-ran `initDb()` end-to-end during a real outage, hammering the DB layer. The new contract: successful inits cache as before; failed inits cache for 30s before the next caller is allowed to re-attempt. In-flight callers still share the same promise. **Departure from plan §5.1.2 documented inline** — the plan's "cache forever for corrupt + 30s TTL" is internally inconsistent; picked the simpler uniform 30s policy because corruption can be repaired externally (operator deletes index.db) and we want to notice on the next retry. `tests/dataInitRetry.test.ts` first case rewritten to pin the new contract: cache held within 30s (initDb 1×) → fake-timer advance past TTL → cache expires (initDb 2×).

- **Wave 1.2 — review-feedback fixes (PR #59 follow-up).** Resolves 11 review comments from Copilot + Codex on PR #59 across the Wave 1.2 work.
  - **P1 — `BackupId` collision-resistant** (`src/lib/configHistory.ts`). Old format `<iso-ts>_<sha8|missing>` could collide for two snapshots in the same millisecond on identical bytes (or any two missing-file snapshots), causing ambiguous restore, React key collisions, and snapshot path aliasing. New format `<iso-ts>_<sha8|missing>_<rand4>` adds 32 bits of randomness — collision probability is negligible. New test case fires 20 concurrent `recordPreWrite` calls on identical bytes and asserts all 20 ids are distinct.
  - **P1 — Snapshot rollback for no-op applies** (`src/lib/template/apply.ts`). `snapshotBeforeApply` ran before the apply primitive decided whether to write. With `conflict: "skip"` (and similar no-op paths), the manifest filled with misleading "restore points" for events that never touched disk. New `removeBackup(id)` API + `finalizeSnapshots(ids, result)` helper roll back the snapshot if the apply result is `skipped` / `would-apply` / `error`. Every dispatcher (agent / skill / command / hook / mcp / workflow / settings / plugin) routes through the new pattern: `const snaps = await snapshotBeforeApply(...); return finalizeSnapshots(snaps, await applyXxx(...));`.
  - **P1 — Prune read-modify-write under one lock** (`src/lib/configHistory.ts`). `prune()` previously read the manifest via `list()`, computed its retention decision, then re-acquired `withFileLock` to rewrite — a concurrent `recordPreWrite` append landing between read and rewrite was silently dropped. The full read+decide+write cycle now lives inside a single `withFileLock` scope.
  - **P2 — Byte-faithful restore** (`src/lib/configHistory.ts` + `src/lib/atomicWrite.ts`). `restore()` decoded the base64 snapshot to a Buffer then converted it to UTF-8 text before writing — corrupting any non-UTF-8 / binary content. `writeFileAtomic` extended to accept `string | Buffer` (raw bytes when given a Buffer, no transcoding); `restore()` now passes the decoded Buffer through unchanged. New test pins this with a fixture containing invalid-UTF-8 bytes (lone surrogates, raw 0xFF, PNG header) and asserts bit-exact equality after restore.
  - **P2 — Catalog + commands cache invalidation on restore** (`src/app/api/config-history/restore/route.ts`). Restore can target agent / skill / command / workflow files but the route only cleared scan/config caches — those views stayed stale until TTL. Added `invalidateCatalogCache()` + `invalidateCommandsRouteCache()` to mirror the apply path's invalidation set.

Template applies to newly-bootstrapped projects route through the `path` branch (post-bootstrap), so their snapshots had `projectSlug: undefined` and were excluded from `/api/config-history?project=<slug>`. New `resolveProjectSlugForSnapshot()` helper looks up the slug from the cached scan when target is path-kind. Returns `undefined` for paths that don't match a scanned project — those still record under the global manifest, just unscoped.

The browser doesn't need a path inside `~/.minder/config-history/`; surfacing it just couples the client to filesystem layout. New test (`tests/configHistoryApi.test.ts`) pins the stripping with a string match against `.minder/config-history` in the serialized response.
  - **P2 — Wrap `list()` in try/catch in restore route** (`src/app/api/config-history/restore/route.ts`). Manifest I/O errors (permissions, FS corruption) escaped as opaque crashes; now surface as structured 500 JSON.
  - **P2 — Fix `shortTs()` for restore toast** (`src/components/ConfigHistoryTab.tsx`). Function received the `BackupId` after restore and tried to reverse its `:` → `-` substitution — the regex didn't account for the format's trailing suffix, so toasts often showed the raw id. ## [1.0.0] - 2026-05-03

> **Headline**: SQLite indexer goes default-on with a fully-indexed read path. Every JSONL-derived dashboard route (`/api/usage`, `/api/sessions{,/[id],/activity}`, `/api/agents{,/[id]}`, `/api/skills{,/[id]}`, `/api/stats`) now serves from `~/.minder/index.db` by default. Fall-throughs to file-parse stay only for v3-readiness windows and brand-new installs; silent error masking is gone.

### Changed
- **P2b-9 — silent fallback removed from the data façade.** Step 2 of the P2b-4 default-on rollout: when DB mode is requested and the DB is genuinely unhealthy (driver missing, schema init failed, connection null, or a SQL load throws), `src/lib/data/index.ts` now THROWS `DbUnavailableError` instead of swallowing the error and serving a file-parse result. Routes get a 500 — which is the right answer for "DB is supposed to work and doesn't." Pre-P2b-9 behavior masked failures via `logFallthroughOnce` and silently downgraded every read for the rest of the dev session, observed in soak as a single transient `EBUSY` poisoning the cached `ensureSchemaReady` (since fixed in P2c).
  - **`MINDER_USE_DB=0` still works** as the explicit file-parse opt-out — operators retain a documented escape hatch for debugging or platform-specific issues. The change only affects the "DB requested + DB broken" path.
  - **Two intentional fall-throughs preserved**, both logged as "expected during X" once per process per scope:
    1. **v3-readiness gate** (`needsReconcileAfterV3`): between migration apply and reconcile, `cost_usd` is 0 on every existing row. File-parse keeps numbers honest during the catch-up window. Without this, the SQL backend would silently serve `totalCost = $0`.
    2. **Empty-index gate** (sessions / agents / skills / claude usage): a brand-new install with the indexer still warming up returns zero rows; falling back keeps the dashboard populated rather than blank during the first scan. Per-session detail miss (session not in index yet) similarly falls back so a newly-arrived JSONL still resolves.
  - **`DbUnavailableError` exported** with a `reason: 'driver-missing' | 'init-failed' | 'connection-null' | 'load-failed'` discriminator and an optional `cause` chain back to the underlying error, so future route-level handling (typed retry, friendlier UI, etc.) can pattern-match without parsing a string. The default route handler doesn't distinguish — Next.js surfaces both as 500 — but the typed shape keeps the contract grep-able.
  - **Test-suite updates**: `tests/dataInitRetry.test.ts` first test flips from "fall-through to file" to "throws DbUnavailableError({reason: 'init-failed'})" while still pinning the retry-on-failure contract (second call must re-attempt `initDb()`). `tests/dataUsage.test.ts` "init-failure" test similarly flips to expect a throw. The other 642 tests are unaffected — the v3-catch-up fallthrough path keeps its existing fall-back behavior, so `tests/dataUsage.test.ts` "falls back when needs_reconcile_after_v3 is set" continues passing as-is. All 642 tests pass, typecheck clean.

### Fixed
- **P2c — three follow-ups from the P2b read-coverage arc.** Ship together as one slice so reviewers can land them in a single PR.
  - **`ensureSchemaReady` doesn't cache one-shot init failures** (`src/lib/data/index.ts`). Original behavior cached the `initDb()` Promise once per process; if the first call hit a transient `EBUSY` on the corruption-quarantine rename (observed in real soak), the cached `{available: false}` poisoned every DB-backed read until process restart, with `logFallthroughOnce` masking repeats. Now: only cache successful inits — clear the cached promise on failure (resolved or rejected) so the next caller retries. Concurrent in-flight callers still share one promise, so there's no thundering-herd risk during init itself; retries only kick in after a failed promise has settled. 2 new tests in `tests/dataInitRetry.test.ts` pin the contract (retry happens, in-flight sharing still works).
  - **`searchableText` restored in DB-backed `/api/sessions`** (`src/lib/data/sessionsListFromDb.ts`). Closes P2b-5 divergence #2 — `SessionsBrowser`'s content-search branch was a no-op under MINDER_USE_DB. New 6th query reads `(session_id, role, text_preview)` for non-empty previews in turn-order; `groupSearchable` builds the same shape as file-parse's `searchParts` accumulator (cap 4000 chars per session, assistant text sliced to 500 per turn, user text untruncated since ingest already capped `text_preview` at 500). Total payload for the typical 150-session corpus is a few MB. Help docs (`docs/help/sessions.md` + runtime copy) reverted to original wording — body-text search works under DB now. Small remaining quirk (file-parse iterates BLOCKS, DB iterates TURNS) covered by detail loader divergence #6; practical impact for content search is negligible.
  - **`sessions.status` populated at ingest** (`src/lib/db/ingest.ts` + `src/lib/data/sessionsListFromDb.ts`). Closes P2b-5 divergence #3 — `needs_attention` no longer silently absent under DB mode. Ingest now walks each parsed session inline with the same pending-pairs algorithm as `inferSessionStatus` (`src/lib/scanner/sessionStatus.ts`) and stores a snapshot in `sessions.status`. The full TypeScript `SessionStatus` (`working|idle|needs_attention`) doesn't fit the existing CHECK constraint and changing the constraint would require a SQLite table rebuild — not worth it for what's effectively a one-bit signal. Snapshot collapses to `'waiting'` (had unresolved pendings) vs `'inactive'` (no pendings). Loader applies time-gating against `file_mtime_ms` at READ time to derive `working / needs_attention / idle` — `working` if mtime fresh (< 90 s), `needs_attention` if in [90 s, 10 min] window, `idle` if older or stored=`'inactive'`. Sessions written before P2c (status=NULL) fall back to the original 2-min mtime heuristic until next full reconcile. **Documented staleness window**: `appendSessionTail` only refreshes status when the tail itself contains an assistant turn; user-only tails (e.g., a tool completion arriving alone) keep the prior `'waiting'` snapshot in place. Time-gating hides this for abandoned sessions (they age out to `idle` after 10 min) but a session that resolved its pendings within the same dev-server lifetime keeps showing as `working` / `needs_attention` until next full reconcile. Acceptable for the typical ~150-session corpus.

No auth on the route — local-only dashboard convention. Validated against the user's real 231 MB DB: `SELECT COUNT(*) FROM sessions` returns in 1 ms, top-projects aggregation across 158 sessions in 1 ms.

Without this, a cold start with no `~/.minder/index.db` would auto-create an empty file via `getDb()` and queries against `sessions` etc. would fall into the 400 "no such table" path instead of the documented 503. Cached as a promise so concurrent first requests share one in-flight init rather than racing migrations.

- **P2a slice 2.3 — byte_offset incremental tail.** A session JSONL that grows (the common case while an agent is working) now reconciles by reading only the new bytes instead of re-parsing the whole file. Existing turn / tool_use / file_edit rows stay put — the tail just appends new ones starting at `MAX(turn_index) + 1` and the session row's aggregates re-derive from the union of old + new turns. On a 50 MB session-in-progress that adds one turn, reconcile work drops from 50 MB → ~1 KB of read.
  - **`reconcileSessionFile` dispatch** — three branches now: (1) gate (file unchanged → 0); (2) tail (same path, derived_version match, size grew, mtime forward → tail-append); (3) full replace (any other change, including file rename, version bump, file shrink, or first-time ingest).

  - **`appendSessionTail`** — INSERTs new rows then UPDATEs the session row's aggregates from old + new combined. Numeric aggregates (token totals, turn counts, error counts, start/end timestamps) re-derive directly in SQL via `SUM` / `MIN` / `MAX` on the turns table. `primary_model` is one indexed `GROUP BY` query. `has_one_shot` re-runs `detectOneShot` over the full UsageTurn[] (rehydrated from DB rows via the new `loadExistingTurnsAsUsage` helper) because the detector's verdict on a prior turn can flip when a verification turn lands in the tail.

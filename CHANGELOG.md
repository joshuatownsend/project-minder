### Fixed
- **Home Live activity card read "(no prompt)" for every session.** Real Claude Code JSONL stores HUMAN-typed user turns with `message.content` as a **string** (e.g. `"fix the build please"`), while assistant turns and tool-result turns use arrays of typed blocks. The DB-ingest pipeline in `src/lib/db/ingest.ts:666` ran `extractText()` from `@/lib/usage/contentBlocks` on whatever `message.content` happened to be — and `extractText()` is intentionally array-only (its docstring says it walks `{type, text}` block lists). For string content it short-circuits to `""`, which meant `userText === ""`, so `isHumanText(userText)` was false, so the `initial_prompt` / `last_prompt` assignment block at lines 670-676 never ran. Every real session in the SQLite index ended up with NULL prompt columns, and `/api/sessions` responded with neither `initialPrompt` nor `lastPrompt` (the fields are optional in `SessionSummary`, so `JSON.stringify` dropped them entirely). Home read `s.lastPrompt?.slice(0, 60) || s.initialPrompt?.slice(0, 60) || "(no prompt)"` and fell through to the empty-state literal for every row. `searchableText` masked the problem because the search-index accumulator IS fed from assistant array content, so the field looked populated even though it never contained the user prompt that anchors the session's identity. File-parse path was unaffected (it uses its own `extractHumanText()` which handles strings). Fix at the DB-ingest call site adds a `typeof textSource === "string" ? textSource : extractText(textSource)` branch — keeps `extractText()`'s array-walking semantics for callers that pass typed block arrays (the documented intent of the helper) and handles the string shape exactly at the user-turn site where Claude Code legitimately emits it. New regression test in `tests/dbIngest.test.ts` builds a session with raw-string `message.content` for two user turns and asserts `initial_prompt` / `last_prompt` land on the row; the pre-existing array-content path is still exercised by the other 24 ingest tests. **Note for existing installs:** the SQLite index keys ingest off JSONL file mtime, so already-indexed sessions stay with NULL prompts until their JSONL file changes (which happens naturally as Claude Code appends new turns to active sessions). To populate prompts for old idle sessions, delete `~/.minder/index.db*` and restart the dev server — the indexer rebuilds from JSONL with the fixed extractor.
- **OTEL receivers bypassed the DB state machine and 500-ed on EBUSY.** Both `src/app/api/otel/v1/metrics/route.ts` and `src/app/api/otel/v1/logs/route.ts` had their own `let initPromise: Promise<…> | null = null` and called `initDb()` from `@/lib/db/migrations` directly. That path is the raw migration runner — no retry, no transient-error classification, no quarantine cap. When the indexer worker held the SQLite file open and an OTEL POST arrived, `getDb()` threw on `new Database(DB_PATH)`, `initDb()` interpreted it as corruption, `quarantineCorruptDb()` tried to `fs.rename(index.db, index.db.corrupt-*)`, the rename hit EBUSY because the file was still locked by the indexer's live handle, the fallback `fs.rm` failed for the same reason, and the original EBUSY rebubbled through `ensureReady` into Next's error path as a 500. Replaced both routes' `initPromise` shim with `ensureSchemaReady()` from `@/lib/data` (a previously-internal helper now exported) — the state-machine path that already classifies EBUSY / EPERM / ENOENT / ENOTEMPTY / SQLITE_BUSY / SQLITE_LOCKED as transient, retries with `[100, 300, 900]ms` backoff (`src/lib/data/index.ts:130,148-155`), caches `transient-failed` for 30 s, and caps quarantines at 2 cumulative runs before going `permanent-failed`. **Why `ensureSchemaReady()` rather than `probeInitStatus()`:** the latter intentionally short-circuits when `MINDER_USE_DB=0` so `/api/health` reports `idle` instead of actively probing (correct for a passive health surface), but OTEL is write-side and needs the DB regardless of which read-path backend is selected — using `probeInitStatus()` here would 503 every OTEL POST under `MINDER_USE_DB=0` (Copilot PR #117 P2 regression). Effect: a momentary file lock now returns 503 ("OTEL storage not available") instead of 500, and lock contentions enter the state machine so Home can observe them.
- **Hydration mismatch on the topbar bell badge.** `AppTopbar` consumes `usePulse()` for the notifications-button `title` and badge-dot, both of which depend on `snapshot.approvalCount + snapshot.pendingSteps`. In dev mode the `/api/pulse` poll can resolve before hydration completes (e.g. while React is still hydrating the SSR HTML), which fires a `setSnapshot` with real counts (242 pending steps in the reported repro) and triggers `Hydration failed because the server rendered HTML didn't match the client` — the SSR frame had `title="No new notifications"` with no badge, the post-poll frame had `title="242 pending — open Status"` with a `<span className="badge-dot">`. Fix mirrors the existing `hydrated` flag pattern in `AppSidebar.tsx:138-139` (originally introduced for HIGH-4 in the 2026-05-10 review): `AppTopbar` now gates `totalAlerts` on a post-mount `hydrated` state so the first client frame always reports zero alerts and matches the SSR HTML; the poll-derived value swaps in on the next render after `useEffect` flushes.

- **P2a slice 2.3 — byte_offset incremental tail.** A session JSONL that grows (the common case while an agent is working) now reconciles by reading only the new bytes instead of re-parsing the whole file. Existing turn / tool_use / file_edit rows stay put — the tail just appends new ones starting at `MAX(turn_index) + 1` and the session row's aggregates re-derive from the union of old + new turns. On a 50 MB session-in-progress that adds one turn, reconcile work drops from 50 MB → ~1 KB of read.
  - **`reconcileSessionFile` dispatch** — three branches now: (1) gate (file unchanged → 0); (2) tail (same path, derived_version match, size grew, mtime forward → tail-append); (3) full replace (any other change, including file rename, version bump, file shrink, or first-time ingest).
  - **`readJsonlSession` parameterized with `{ fromOffset, startTurnIndex }`** — when `fromOffset > 0`, reads from the position immediately after the last consumed `\n` (the invariant carried by `sessions.byte_offset` since P2a-2.1) and assigns turn indexes starting from `startTurnIndex` so they don't collide with existing rows.
  - **`appendSessionTail`** — INSERTs new rows then UPDATEs the session row's aggregates from old + new combined. Numeric aggregates (token totals, turn counts, error counts, start/end timestamps) re-derive directly in SQL via `SUM` / `MIN` / `MAX` on the turns table. `primary_model` is one indexed `GROUP BY` query. `has_one_shot` re-runs `detectOneShot` over the full UsageTurn[] (rehydrated from DB rows via the new `loadExistingTurnsAsUsage` helper) because the detector's verdict on a prior turn can flip when a verification turn lands in the tail.
  - **Schema migration v2** adds `turns.tool_result_preview` so the rehydration of user-with-tool-result turns survives a tail-append round-trip. Without it, `detectOneShot`'s error-pattern check (`FAIL`, `Error:`, etc.) saw `undefined` for prior failed verifications, and `has_one_shot` could flip to true on a sequence that had actually failed. `DERIVED_VERSION` bumped to 2 so existing rows re-parse on first reconcile after upgrade and the column is populated before any tail relies on it. v2 migration is idempotent — fresh DBs run the latest `schema.sql` (column already present) and skip the `ALTER TABLE` via a `PRAGMA table_info` check.
  - **Ranged tail read** uses `createReadStream({ start })` instead of `fs.readFile + Buffer.subarray`. The OS only delivers bytes after the offset; an active 50 MB session log now costs ~1 KB of read per appended turn instead of re-reading the whole prefix.
  - **`last_prompt` fallback fixed** — when a tail contains no new human prompt, we keep the existing `last_prompt` instead of regressing it to `initial_prompt`.
  - **Mid-flush safety: `byte_offset` never advances past a partial line.** Both full-parse and tail-parse paths now read up to the LAST `\n` byte and park the cursor there. Anything after the last newline is treated as a partial line that hasn't been flushed yet — the next reconcile picks it up once the writer finishes. Without this, a watcher firing on a half-written line would silently drop that turn forever.
  - **Truncated `arguments_json` recovery on rehydrate.** Bumped `ARGS_JSON_LIMIT` from 10 KB to 32 KB and added a best-effort `command` extraction in `parseStoredArgs`.

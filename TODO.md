## Memory Observatory

*Inspired by [Nicolas Bustamante's "Agent Memory Engineering"](https://x.com/nicbstme/status/2050301124314563025), comparing how Claude Code, Codex CLI, and Hermes implement agent memory. Key findings: clever architectures (vector DBs, knowledge graphs, dedicated memory agents) all lost to **LLM + typed markdown + bash**; the interesting axis is the read/write **discipline** the agent follows, not the data structure; memory is a **hint surface, not authority** — every body read should be wrapped in an age-aware "verify before recommending" reminder. Phase 1 (Freshness + Telemetry + Budget + MEMORY.md index awareness on `/memory`) is in active planning per the approved plan at `~/.claude/plans/i-recently-read-this-temporal-crane.md`; the entries below are the deferred follow-on phases, in priority order.*

The plan doc and this TODO entry are intentionally redundant on the M.2 spec itself; the plan doc adds the *why* (article framing, cross-wave dependency story) that's not duplicated here. Addresses the article's central frustration: memory rules taught to Claude Code don't transfer to Codex / Gemini despite sharing the same filesystem, because each model is post-trained on its harness's specific memory layer (Claude Code's typed taxonomy + always-loaded MEMORY.md index + age-aware `<system-reminder>` framing on every body read vs Codex's offline extract+consolidate pipeline + lazy `memory_summary.md` load + on-demand handbook reads). Extend the existing adapter registry from session-read to full memory IO, then ship a diff/sync UI so canonical rules can be authored once and pushed across stores. **Upstream research needed before PLAN**: (a) **Codex memory layout** — per the article, Codex's memory folder has its own `.git/`, a `memory_summary.md` (5K-token cap, injected per turn), a full `MEMORY.md` handbook, and a `skills/` tree; discover exact path under `$CODEX_HOME/`; document the `MemoriesUsageKind` enum values (`MemoryMd`, `MemorySummary`, `RawMemories`, `RolloutSummaries`, `Skills`) so the bridge can expose per-kind usage.
